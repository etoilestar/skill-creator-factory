# 交互式 Skill 创建指南

本文档用于指导生产者如何用少量问题完成需求挖掘、蓝图确认和测试迭代，尤其适配 4o-mini、32B 小模型等不适合长篇复杂指令的场景。

---

## 1. 每次只问一个问题

不要一次性问：

```text
请告诉我输入、输出、格式、风格、依赖、使用场景、测试方式。
```

应该按顺序问：

```text
好的，我先确认一个关键信息：用户输入什么，它应该输出什么？
```

用户回答后，再问下一个问题。

---

## 2. 优先问真实使用例子

真实例子比抽象描述更可靠。

推荐问题：

```text
这个 Skill 最常见的使用场景是什么？请给我一个用户可能会说的真实例子。
```

---

## 3. 用户模糊时要复述理解

如果用户说：

```text
做一个像某某那样的 Skill。
```

不要直接创建。

应该回复：

```text
好的，我先按我的理解确认一下：你希望它主要完成 <你的理解>，输出 <你的理解>，对吗？
```

---

## 4. 技术方案要用简单语言

不要问：

```text
你要 REST API 还是 GraphQL？
```

应该问：

```text
我理解有两个方案：

A. 直接处理本地文件
- 优点：简单
- 限制：不能自动获取在线数据

B. 连接在线服务
- 优点：功能更完整
- 限制：需要网络和配置

你更倾向哪种？
```

---

## 5. 能力调度要讲人话

不要问：

```text
你需要 planner/coder/writer/verifier 组合吗？
```

应该问：

```text
这个 Skill 是只让当前 AI 助手直接完成，还是需要调用更适合的能力，比如写代码模型、写作模型、画图工具、文件处理脚本或外部服务？
```

---

## 6. 蓝图必须短

蓝图不是完整设计文档。

蓝图只说明：

- 输入
- 输出
- 触发
- 能力分工
- 目录结构
- 资源职责
- 依赖配置
- 测试方式

不要在蓝图阶段输出文件内容。

---

## 7. 未确认蓝图前禁止输出文件

以下行为都不允许：

- 输出完整 `SKILL.md`
- 输出 `scripts/main.py`
- 输出 references 内容
- 输出 assets 内容
- 输出测试命令
- 说已经创建完成

如果用户还没确认，最多只能问问题或输出蓝图。

---

## 8. 测试问题设计

创建完成后，至少设计三类测试：

### 正常请求

应该触发 Skill。

示例：

```text
帮我写一个关于勇敢小狐狸的童话。
```

### 边缘请求

复杂或特殊输入。

示例：

```text
帮我写一个悬疑风格的儿童故事，但结尾要温暖，控制在 300 字以内。
```

### 不应触发

相似但不相关的请求。

示例：

```text
帮我分析狐狸的生物学特征。
```

---

## 9. 迭代优化

如果用户说测试结果不好，先定位问题：

- 没触发：检查 `description`。
- 误触发：收窄 `description`。
- 输出格式不对：检查 `SKILL.md` 的输出要求。
- 脚本失败：检查命令、路径、依赖、输入方式。
- 资源没用上：检查 `SKILL.md` 是否说明何时读取 references/assets。
- 专用能力不可用：检查能力调度说明和降级方案。
- 外部接口失败：检查配置、环境变量和错误处理。

---

## 10. 小模型友好表达

对于小模型，指令应：

- 短句。
- 明确。
- 不嵌套太深。
- 不一次性要求太多。
- 使用固定输出格式。
- 多给正反例。
- 状态机规则放在前面。

避免：

- 长篇抽象原则。
- 多层列表嵌套。
- 模糊措辞。
- “尽量”“最好”这类弱约束过多。

---

## 11. 推荐提问顺序

1. 用户输入什么，输出什么？
2. 典型使用场景是什么？
3. 输出形式是什么？
4. 是否需要格式、风格、模板或领域规范？
5. 是否需要脚本、文件处理、外部接口、专用模型或工具？
6. 是否允许第三方库、环境变量或外部服务？
7. 是否给自己用、偶尔用、经常用、团队用？
8. 蓝图是否确认？
9. 确认后再创建。
