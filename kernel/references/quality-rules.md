# Skill 质量规则与反空心实现标准

本文档用于指导 Skill 生产者判断生成的 Skill 是否真正可用。

---

## 1. 什么是可用 Skill

一个可用 Skill 至少应该满足：

1. 用户知道什么时候使用它。
2. Agent 知道什么时候触发它。
3. Agent 知道输入需要包含什么。
4. Agent 知道按照什么流程处理。
5. Agent 知道输出应该长什么样。
6. 缺少信息时知道如何追问。
7. 有领域规则、模板或脚本时，能找到对应资源。
8. 有脚本时，脚本能真实运行。
9. 有依赖时，依赖被声明。
10. 有外部接口时，配置方式被说明。
11. 有专用模型能力时，有输入输出契约和降级方案。
12. 不编造执行结果。
13. 不用空心逻辑冒充能力。

---

## 2. 什么是空心 Skill

空心 Skill 指看起来有文件、有脚本、有说明，但实际上没有完成任务能力的 Skill。

典型表现：

```python
return data
```

```python
return f"结果：{data}"
```

```python
return {"result": data}
```

```python
# 这里简单处理
```

```python
# 实际业务中可扩展
```

这类实现只是在包装输入，没有形成真实能力。

---

## 3. 禁止的空心表达

生成 Skill 时，禁止出现：

- `pass`
- `TODO`
- `待实现`
- `这里简单处理`
- `实际业务中可扩展`
- `此处省略`
- `根据需要补充`
- `简单返回输入`
- `模拟生成`
- `伪代码如下`
- `这里只是示例`

除非用户明确要求写教学示例，否则这些内容都不应出现在最终 Skill 包中。

---

## 4. 脚本质量标准

如果创建 `scripts/`，脚本必须承担真实、可验证的职责。

合格脚本职责示例：

- 字段校验
- JSON/YAML/CSV 转换
- 模板渲染
- 文件处理
- 批量处理
- 生成 Word / HTML / Markdown / PDF 文件
- 调用 API
- 调用本地模型或外部模型
- 调用数据库或检索服务
- 保存文件并返回路径

不合格脚本职责示例：

- 只返回用户输入
- 只给输入加前缀
- 只把输入包装成 JSON
- 用固定片段拼接故事
- 用随机片段伪装高质量创作
- `SKILL.md` 中说运行脚本，但脚本文件不存在

---

## 5. description 质量标准

`description` 是触发依据，必须清楚。

不合格示例：

```yaml
description: 用于处理文本。
```

```yaml
description: 一个工具。
```

合格示例：

```yaml
description: 根据用户提供的主题、角色、风格和篇幅要求生成中文故事，并可按需导出为 Word 文档。当用户要求写故事、生成童话、创作短篇故事或把故事保存为文档时使用。
```

好的 description 应包含：

1. 做什么。
2. 输入大致是什么。
3. 输出大致是什么。
4. 什么时候触发。
5. 必要时说明不适用场景。

---

## 6. SKILL.md 质量标准

`SKILL.md` 不应该太长，但必须可执行。

至少应包含：

```markdown
## 输入要求

## 工作流程

## 能力调度说明

## 资源使用说明

## 输出要求

## 依赖与配置

## 失败处理
```

如果不需要能力调度、依赖或配置，可以写“无”。

要求：

1. 工作流程必须是可执行步骤。
2. 输出要求必须明确。
3. 失败处理必须说明缺什么时如何追问。
4. 如果有 resources，必须说明何时读取或使用。
5. 如果有专用模型能力，必须说明输入输出和降级方案。
6. 不要在 `SKILL.md` 中堆长篇知识，长内容放入 `references/`。

---

## 7. 脚本与 SKILL.md 一致性

如果 `SKILL.md` 中写了脚本调用方式，必须保证：

1. 脚本文件真实存在。
2. 命令中的路径与实际文件路径一致。
3. 脚本接收输入的方式与文档一致。
4. 脚本输出的格式与文档一致。
5. 脚本失败时给出中文错误或清晰错误。
6. smoke test 能运行。

错误示例：

```markdown
运行 python scripts/main.py
```

但目录中没有 `scripts/main.py`。

错误示例：

```markdown
运行 python scripts/main.py --input "..."
```

但脚本只从 stdin 读取，不支持 `--input`。

---

## 8. 外部接口质量标准

如果 Skill 使用外部接口，必须说明：

1. 接口用途。
2. 需要哪些环境变量。
3. 需要哪些配置。
4. 输入输出格式。
5. 失败时如何处理。
6. 不得写死密钥、Token、密码。

错误示例：

```python
API_KEY = "sk-xxxx"
```

正确做法：

```text
从环境变量 LLM_API_KEY 读取。
```

---

## 9. 模型能力调度质量标准

如果 Skill 需要专用模型能力，必须说明：

1. 能力名称。
2. 调用条件。
3. 输入是什么。
4. 输出是什么。
5. 结果如何被后续步骤使用。
6. 能力不可用时如何降级。

错误示例：

```text
调用写作模型生成故事。
```

正确示例：

```text
如果宿主提供 writing 能力，将主题、角色、风格和篇幅传给写作模型生成正文；如果 writing 能力不可用，则当前模型根据 references/story-structure.md 直接生成正文。
```

---

## 10. 校验清单

创建 Skill 后必须检查：

- [ ] `SKILL.md` 是否存在。
- [ ] frontmatter 是否合法。
- [ ] `name` 是否使用小写字母、数字和连字符。
- [ ] `description` 是否清楚。
- [ ] 是否包含输入要求。
- [ ] 是否包含工作流程。
- [ ] 是否包含能力调度说明。
- [ ] 是否包含资源使用说明。
- [ ] 是否包含输出要求。
- [ ] 是否包含依赖与配置。
- [ ] 是否包含失败处理。
- [ ] 是否存在空心脚本。
- [ ] 如果有脚本，脚本是否真实存在。
- [ ] 如果有脚本，调用方式是否一致。
- [ ] 如果有脚本，是否给出 smoke test。
- [ ] 如果有 references/assets，SKILL.md 是否说明用途。
- [ ] 如果有依赖，是否声明。
- [ ] 如果有外部接口，是否说明配置。
- [ ] 如果有专用模型能力，是否说明降级方案。
- [ ] 是否有最小验收样例。

---

## 11. 失败时如何处理

如果校验失败，不要直接报告完成。

应该：

1. 说明发现的问题。
2. 修改对应文件。
3. 重新校验。
4. 再报告结果。

不要假装测试通过。
